# Phase 0 go/no-go verdict — 2026-07-03

**Verdict: GO** — gate conditions 1 (accuracy) and 3 (memory headroom) pass
outright. Gate condition 2 as written in §10 requires the airplane-mode test,
which was **deferred to Phase 3** (issue #14); it is satisfied here only by the
weaker egress-sampling evidence below. The GO is issued on that explicit,
reduced basis, with one §11 performance KPI additionally flagged amber
(issue #13; details below). Data: `report.md` +
`results.json` in this directory (30 synthetic letters, seed 42; generated by
`testset/`, scored by `spike/benchmark.py` on the production host — the 8 GB M1
Mac mini, with Plex/Sonarr etc. resident as in real life).

## Gate condition 1 — accuracy targets: PASS

| Metric | Target (§11) | Measured | |
|---|---|---|---|
| OCR char accuracy, well-lit | ≥ 98% | **99.9%** (Apple Vision, clean+good) | ✅ |
| OCR char accuracy, average | ≥ 95% | **99.9%** (poor tier) | ✅ |
| doc_type | ≥ 95% | **100%** | ✅ |
| sender/correspondent | ≥ 90% | **100%** | ✅ |
| document date | ≥ 95% | **100%** | ✅ |
| amount / IBAN / reference | ≥ 98% | **100% / 100% / 100%** | ✅ |
| tag precision (controlled vocab) | ≥ 90% | **100%** (after §6.4 reconciliation) | ✅ |

Caveat (issue #6): synthetic letters are an optimistic stand-in — single-column,
one font, simulated degradation. **These numbers are upper bounds.** The same
benchmark re-runs on real letters via `data/testset-real/` (see
`testset/README.md`); that re-run gives the definitive numbers.

OCR head-to-head: Apple Vision 99.87/99.92/99.90% per tier vs PaddleOCR
(PP-OCRv5 latin, mobile det) 95.43/94.67/92.53% — **Apple Vision is primary**
(decision log #6). PaddleOCR also required two env vars to stop it phoning
home (paddlex source check + hf-hub cache revalidation) — both handled in
`spike/ocr_engines.py`, both reasons it stays the alternate.

## Gate condition 2 — no cloud dependence: egress-sampling evidence only (formal test deferred)

- The egress monitor (2 s sampling of all pipeline processes for established
  non-loopback TCP) recorded **zero** connections across the full record run.
- All models local and cached; the pipeline makes exactly one kind of network
  call: HTTP to Ollama on 127.0.0.1.
- The **formal airplane-mode test** (physically cutting the host's uplink) is
  deliberately deferred to the Phase 3 acceptance gate: this host also runs the
  owner's media stack, and killing its network mid-execution was not
  acceptable. The §11 privacy KPI stays open until that test passes.

## Gate condition 3 — memory headroom on the 8 GB host: PASS

- Peak sampled RSS of the pipeline processes: **~1.0 GB**, plus the VLM's
  **3.6 GB** resident in Metal-shared memory (`ollama ps`), alongside the
  owner's media services. No swap collapse, no OOM, across 90+ letters of
  cumulative benchmarking. Note: one KV-cache misstep destroys this — `num_ctx`
  must stay 4096 (decision log #8).

## §11 performance KPI — AMBER (not a Phase 0 gate condition)

- Production-config pipeline per page ≈ **~21 s** (measured means: preprocess
  0.4 s + Apple Vision OCR 1.5 s + extraction median 18.9 s / mean 21.9 s —
  see `report.md`) vs the **≤ 15 s** KPI.
- Measured under benchmark conditions with PaddleOCR also running per letter;
  the cleanest earlier run showed extraction medians ~14–16 s. Capture UX is
  unaffected (processing is async; phone upload returns immediately).
- Paths to green, in preference order: (a) `qwen3-vl:2b-instruct` (~2× faster,
  accuracy to be benchmarked), (b) trim the extraction schema/prompt,
  (c) future hardware. Tracked as follow-up issue #13.

## What changed because of this spike

Decision log v0.3 in plan.md (#6–#9): Apple Vision primary; text-first
extraction (image input measurably hurt on 4B models); `-instruct` tag +
`num_ctx` 4096 + required-nullable schema pins; deterministic tag
reconciliation. Two truth-data fixes came out of the benchmark itself (dentist
pool, `bill` tags) — the extractor caught a labeling error, which is exactly
the §12 loop working.
